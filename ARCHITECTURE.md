## 1. Extraction Pipeline (Deep Dive)

The extraction pipeline is the most critical and computationally expensive component. It runs once per book and produces the World Bundle. Quality here determines everything downstream.

### Multi-Pass Design

Each pass is isolated, cacheable, and versioned. If pass 2 fails or needs re-running, you don't re-run pass 1.

```
Pass 1: STRUCTURE
  Input:  Cleaned, segmented text (chunks of ~2000 tokens)
  Output: entities.json, locations.json, objects.json, events.json
  Cost:   ~1 LLM call per chunk

Pass 2: RELATIONS
  Input:  entities.json + relevant text chunks
  Output: social_graph.json, conflicts.json, timeline.json
  Cost:   ~1-2 calls per character pair

Pass 3: PSYCHOLOGY
  Input:  character list + all dialogue/action passages per character
  Output: psychology/{character_id}.json
  Cost:   ~3-5 calls per major character (most expensive)

Pass 4: SYMBOLISM
  Input:  Full text summary + entities + events
  Output: themes.json, motifs.json, world_rules.json, narrative_gravity.json
  Cost:   ~5-10 calls total
```

### Chunking Strategy

Books are not processed linearly. The segmenter produces:
- **Micro-chunks**: ~500 tokens — for entity extraction
- **Scene-chunks**: ~2000 tokens — for event/relation extraction  
- **Character-chunks**: All passages containing a given character — for psychology pass
- **Full-book summary**: 2000-token compressed overview — for symbolism pass

### Caching

Every LLM output is cached by `hash(prompt + chunk_id)`. Re-running extraction after a model swap only re-processes uncached items. This makes iteration on prompts cheap.

Each pass produces versioned JSON. Example from Pass 3:

## 2. World Bundle Format

The compiled world is a single serializable bundle. It is the persistent source of truth for the simulation.

```
world.bundle/
├── manifest.json          # Version, source book, compilation timestamp
├── locations.json         # Full location graph
├── characters.json        # Character models (no psychology — separate file)
├── psychology/
│   └── {char_id}.json     # Per-character full psychology
├── objects.json
├── events_canon.json      # Original book timeline
├── social_graph.json
├── world_rules.json
├── narrative_gravity.json # Canon event weights
└── agents/
    └── {char_id}/
        ├── system_prompt.txt
        └── initial_memory.json
```

Serialization format: JSON for human-readability during development, MessagePack for production (faster I/O).

## 3. Memory Architecture

The simulation uses a three-layer memory system. This is essential — you cannot fit an entire session into a single LLM context.

### Layer 1: Structured State (Source of Truth)

```
world/state.json
```

Pure data. No LLM involvement. Contains:
- Current location of every character and object
- Current value of every world variable (relationship scores, event flags, etc.)
- Tick counter
- Divergence score

This is loaded fresh on every interaction. It is never generated by the LLM — only updated by the game engine based on validated actions.

### Layer 2: Vector Memory

Each agent and the world narrator has a FAISS index. Events are embedded and stored. On each LLM call, the top-k most relevant past events are retrieved and injected into context.

```python
# Per-agent memory query
relevant_memories = agent.memory.query(
    query=current_scene_description,
    k=5,
    filter={"recency_weight": 0.3}
)
```

### Layer 3: Active Context Window

Built fresh for each LLM call. The `ContextBuilder` assembles:
1. Agent system prompt
2. World state snapshot (current location, present characters/objects)
3. Retrieved memories (from vector store)
4. Recent conversation history (last N turns)
5. Current action to evaluate

Target: stay under 4096 tokens for fast inference on local hardware.

### Memory Compression

When an agent's episodic memory grows large, the `Summarizer` runs:
- Groups events by scene
- Produces compressed "memory chunks" (e.g. "During the Council of Elrond, you argued for the Ring's destruction and Frodo volunteered as bearer")
- Original events archived, compressed version kept in active memory

## 4. Decision Engine

Agents don't just respond to player input — they act autonomously during ticks.

### Goal Evaluation

On each tick, each active agent runs:

```python
def evaluate_tick(agent, world_state):
    # 1. What do I know about the current state?
    context = build_agent_context(agent, world_state)
    
    # 2. Which of my goals are currently actionable?
    actionable_goals = [g for g in agent.goals if is_reachable(g, world_state)]
    
    # 3. Pick highest-priority actionable goal
    top_goal = max(actionable_goals, key=lambda g: g.priority)
    
    # 4. Ask LLM: what action would I take right now?
    action = llm.decide(agent.system_prompt, context, top_goal)
    
    # 5. Validate + apply
    return action_executor.apply(action, world_state)
```

### Tick Frequency

Not all agents tick equally. The scheduler uses three tiers:

```yaml
tick_tiers:
  primary:    # Major characters, ticks every turn
    frequency: 1
    examples: [protagonist, main_antagonist]
  
  secondary:  # Supporting characters, ticks every N turns
    frequency: 3
    examples: [major_supporting]
  
  background: # Minor characters, lazy evaluation
    frequency: 10
    examples: [unnamed_guards, shopkeepers]
```

The Author Ghost ticks every turn at the lowest priority — after all characters have acted.

## 5. Narrative Gravity System

Narrative gravity is a quantified pull toward canonical events. It operates at two levels:

### Event Gravity

Each canonical event has a `gravity_weight` (0.0–1.0). High-gravity events are nearly impossible to prevent; low-gravity events are easily diverged from.

```json
{
  "event_id": "frodo_leaves_shire",
  "gravity": 0.90,
  "triggers": ["frodo_has_ring", "gandalf_warns_frodo"],
  "description": "Frodo departs the Shire with the Ring"
}
```

During simulation, the engine checks: are gravity conditions met? If yes, the Author Ghost begins nudging agents toward the event.

### Agent Gravity

Individual characters have a `canon_adherence` personality trait. High-canon characters (Gandalf: 0.80) resist player manipulation more strongly than low-canon ones (random innkeeper: 0.10).

### Divergence Tracking

The simulation tracks a `divergence_score` — how far the current timeline has strayed from canon. High divergence:
- Reduces Author Ghost effectiveness
- Unlocks "fully diverged" mode (sandbox, all gravity disabled)
- May trigger unique non-canonical events

## 6. The Author Ghost

The Author Ghost is a special agent that has no physical presence in the world. It cannot be interacted with. It acts through subtle modifications to other agents' decisions.

### Mechanism

After all regular agents have ticked, the Author Ghost evaluates:

```python
def ghost_tick(ghost, world_state, agent_decisions):
    # 1. Check: are any gravity events at risk?
    at_risk = get_endangered_canonical_events(world_state)
    
    # 2. For each at-risk event, identify the agent most able to restore it
    for event in at_risk:
        target_agent = find_best_nudge_target(event, world_state)
        
        # 3. Generate a subtle nudge — not a command, an inclination
        nudge = ghost.generate_nudge(target_agent, event)
        
        # 4. Inject as a weighted bias into that agent's next decision
        target_agent.add_temporary_bias(nudge, weight=event.gravity * 0.3)
```

Nudges are invisible to the player but create believable character behavior. Frodo "feels compelled" to move toward the fire. Gandalf "senses urgency" to arrive in time.

## 7. LLM Client Architecture

All LLM calls go through a unified interface. This lets you swap backends (llama.cpp → vLLM → API) without touching simulation code.

```python
class LLMClient:
    def complete(self, prompt: str, max_tokens: int, temperature: float) -> str: ...
    def embed(self, text: str) -> List[float]: ...

class LlamaCppClient(LLMClient):
    # Connects to local llama.cpp server (--server mode)
    def complete(self, prompt, max_tokens=512, temperature=0.7) -> str: ...
```

### Recommended Models (by task)

| Task | Model | Notes |
|---|---|---|
| Extraction (pass 1-2) | Mistral-22B-Instruct | Good JSON output |
| Psychology extraction | Mistral-22B or Qwen2.5-32B | Needs nuance |
| Agent narration | Mistral-7B or Llama-3.1-8B | Fast, good prose |
| Decision engine | Mistral-7B | Speed matters here |
| Author Ghost | Mistral-22B | Needs subtlety |

Using smaller models for fast tasks (agent decisions, narration) and large models for slow tasks (extraction, psychology) is the recommended setup.

## 8. State Persistence

The world state is snapshotted after every player action and after every tick.

```
data/compiled/alice/sessions/
├── session_20241201_143200/
│   ├── state_tick_000.json
│   ├── state_tick_001.json
│   ├── ...
│   └── session_meta.json
```

### Save/Load

Any tick snapshot can be loaded as a save point. Sessions are self-contained — you can share a session folder and resume on any machine.

### Branching

The architecture naturally supports branching timelines: duplicate a snapshot folder, diverge from that point. This could support a "what if?" mode in a future version.

## 9. Interaction Parser

Player input is free-form natural language. The parser converts it to structured intents.

### Intent Schema

```python
Intent {
    action: str          # "move", "talk", "take", "examine", "attack", "wait"
    target: Optional[str]
    target_type: str     # "character", "location", "object"
    parameters: Dict
    raw_input: str
    confidence: float
}
```

### Parsing Strategy

Hybrid approach — rule-based fast path for common commands, LLM fallback for ambiguous input:

```python
def parse(raw_input: str, world_state: WorldState) -> Intent:
    # Fast path: known command patterns
    if match := COMMAND_PATTERNS.match(raw_input):
        return build_intent_from_match(match, world_state)
    
    # Slow path: LLM interpretation
    return llm_parse(raw_input, world_state.current_context())
```

## 10. Extension Points

StoryWeaver is designed to be modded. Key extension points:

### Custom World Rules

```python
# plugins/combat_system/rules.py
class CombatRule(WorldRule):
    def applies(self, action: Intent) -> bool:
        return action.action == "attack"
    
    def resolve(self, action: Intent, world_state: WorldState) -> List[Event]:
        ...
```

### Custom Personality Templates

```yaml
# agents/prompts/personality_templates/antihero.yaml
name: Antihero
description: Selfish but ultimately redeemable
traits:
  agreeableness_modifier: -0.3
  courage_modifier: +0.2
  deceptiveness_modifier: +0.4
prompt_injection: |
  You act in your own interest first. You help others only when it serves you —
  or when you can't help yourself.
```

### Custom Extraction Prompts

All extraction prompts are external `.txt` files. Swap them to tune extraction behavior without code changes.
